### Definition 2.1: Set Union Ratio (Ој)

> **Ој(Г…) = |вҲӘ_{AвҲҲГ…} A| / ОЈ_{AвҲҲГ…} |A|**

This maps a set collection to a rational number in [0,1].

**Example:** For Г… = {{1}, {1,2}, {1,2,3}}:
```
|вҲӘ A| = |{1,2,3}| = 3
ОЈ |A| = 1 + 2 + 3 = 6
Ој(Г…) = 3/6 = 1/2
```

### Definition 2.2: Binary Encoding Rule (ПҮбөў)

> **ПҮбөў(Г…) = 1 if i = n + |AвӮҷ| - 1 for some AвӮҷ вҲҲ Г…**
> **ПҮбөў(Г…) = 0 otherwise**

This generates a binary string from the set collection.

**Example:** For Г… = {{1}, {1,2}, {1,2,3}}:
```
AвӮҖ: n=0, |AвӮҖ|=1  вҶ’  i = 0 + 1 - 1 = 0  вҶ’  ПҮвӮҖ = 1
AвӮҒ: n=1, |AвӮҒ|=2  вҶ’  i = 1 + 2 - 1 = 2  вҶ’  ПҮвӮӮ = 1
AвӮӮ: n=2, |AвӮӮ|=3  вҶ’  i = 2 + 3 - 1 = 4  вҶ’  ПҮвӮ„ = 1

Binary string: 1 0 1 0 1
```

### Definition 2.3: Binary Decoder (D)

The inverse operation recovers a set collection from a binary string:

> For position i where bбөў = 1, solve i = n + |AвӮҷ| - 1 to find n and |AвӮҷ|

**Example:** For binary string `10101`:
```
i=0: 0 = n + |AвӮҷ| - 1  вҶ’  n=0, |AвӮҖ|=1  вҶ’  AвӮҖ = {1}
i=2: 2 = n + |AвӮҷ| - 1  вҶ’  n=1, |AвӮҒ|=2  вҶ’  AвӮҒ = {1,2}
i=4: 4 = n + |AвӮҷ| - 1  вҶ’  n=2, |AвӮӮ|=3  вҶ’  AвӮӮ = {1,2,3}

D(10101) = {{1}, {1,2}, {1,2,3}}
```

### Core Concepts

#### ОЁбөў (Psi_i) вҖ” The Polynomial Sequence

**ОЁбөў** represents a polynomial P at iteration i. The Java code represents this as `pArray`:

```
ОЁвӮҖ = P(x)           вҖ” original polynomial
ОЁвӮҒ = О”P(x)          вҖ” first difference

...

```

From `LoopList.java`:
> *"Psi_i represented as pArray is associated with a collection of LoopList, each a list of DeltaDelta_j values."*

#### TriagTriag (Triangle of Triangles)

**TriagTriag** is the triangular difference table вҖ” the structure formed by iterated differences:

```
TriagTriag for P(x) = xВі:

k:     0    1    2    3    4    5    6
P(k):  0    1    8   27   64  125  216     вҶҗ ОЁвӮҖ

```

Each row is a **TriagTriag_j** вҖ” the jth level of differences.

#### DOSPT (Difference of Scalars Polynomial Tree)

**DOSPT** is a tree structure where:

- **Vertical branches:** Represent TriagTriag levels (j = 1, 2, ..., degree)
- **Horizontal branches:** Represent polynomial transformations Psi_i вҶ’ Psi_{i+1}
- **Nodes:** Each node has coordinates (horizontal_n, vertical_m)

From the notes:
> *"The structure of DOSPT is composed of vertical and horizontal branches rooted in Psi_i. Vertical expansion of a root corresponds to nodes each representing one of its TriagTriag's polynomials."*

#### The Diophantine Connection

The difference structure satisfies **linear Diophantine equations**:

```
ax! + bx! = c(x+1)!
```

From `FIDFormulas.odt`:
> *"TriagTriag is a linear Diophantine Equation ax+by=c. It happens that x=-y and is derived from the factorial at nMax-i."*

This means consecutive TriagTriag levels are algebraically related through factorial relationships.

## Node Classification Task: Determined vs. Undetermined

### The Classification Logic (from code)

From `GaussTable1.java` (lines 913-915):

```java
int determined = 0;
if (dimension - wNum == totalZero) { 
    determined = 1; 
}
```

### What This Means Mathematically

| Variable | Meaning |
|----------|---------|
| `dimension` | Original polynomial degree (e.g., 5 for degree-5) |
| `wNum` | Difference level (0 = original, 1 = 1st difference, 2 = 2nd difference, ...) |
| `dimension - wNum` | **Expected degree at this level** |
| `totalZero` | Count of integer roots found in evaluation range |

**A polynomial at difference level `wNum` is DETERMINED if:**
```
totalZero == dimension - wNum
```

### The Mathematical Insight

**Fundamental Theorem of Algebra:** A polynomial of degree d has exactly d roots (counting multiplicity, in в„Ӯ).

At each difference level:
- The polynomial has degree `dimension - wNum`
- It can have **at most** `dimension - wNum` integer roots in any finite range
- If we find exactly that many zeros вҶ’ the polynomial is **fully characterized** by its roots
- If we find fewer вҶ’ some roots are **outside** the evaluation range or **non-integer**

```
Original (wNum=0):     degree = dimension     вҶ’ max roots = dimension
1st diff (wNum=1):     degree = dimension-1   вҶ’ max roots = dimension-1
2nd diff (wNum=2):     degree = dimension-2   вҶ’ max roots = dimension-2
...
Final (wNum=dimension): degree = 0 (constant) вҶ’ max roots = 0
```

### Why Determined Matters

**Determined (determined=1):**
- Found ALL possible integer roots within the range
- The polynomial's behavior is **fully captured** by `muList`
- Predictable structure вҖ” no hidden roots outside `integerRange`

**Undetermined (determined=0):**
- Found FEWER roots than the degree allows
- Missing roots fall into distinct categories (see detailed analysis below)
- Polynomial has **hidden structure** not captured in `muList`

### Complete Analysis: Why Polynomials Are Undetermined

A degree-d polynomial can have at most d roots (Fundamental Theorem of Algebra). When `totalZero < d`, the "missing" roots fall into the following categories:

#### Category 1: Roots Outside the Observable Range

**Cause:** Integer roots exist but lie beyond `[-integerRange/2, +integerRange/2)`

**Example:** P(x) = (x - 200)(x - 300) with `integerRange = 200`
- True roots: {200, 300}
- Observable range: [-100, +99]
- Detectable: none
- `totalZero = 0`, degree = 2 вҶ’ **undetermined**

#### Category 2: Positive/Negative Definite вҖ” No Real Roots

**Cause:** Polynomial never crosses zero вҖ” all values have the same sign.

**Calculus criterion:** A polynomial P(x) is **positive definite** if P(x) > 0 for all x вҲҲ в„қ.

This occurs when:
- All roots are complex (non-real conjugate pairs)
- For even degree with positive leading coefficient, no real critical points where P = 0

**Examples:**
```
P(x) = xВІ + 1           Always > 0, roots at Вұi
P(x) = xвҒҙ + xВІ + 1      Always > 0, no real roots
P(x) = (xВІ + 1)ВІ        Always > 0, double roots at Вұi
P(x) = xвҒҙ + 4           Always > 0, roots at (1Вұi)вҲҡ2, (-1Вұi)вҲҡ2
```

**Negative definite:** P(x) < 0 for all x вҲҲ в„қ (e.g., `-xВІ - 1`)

**Detection in graph:** At every evaluation point k, P(k) вү  0
- `totalZero = 0` for the affected difference level
- **Always undetermined** regardless of range

**Note:** Due to differentiation properties, if P(x) is positive definite:
- First difference О”В№P may cross zero (not definite)
- Higher differences may or may not be definite
- Definiteness is NOT preserved through differences

#### Category 3: Complex Conjugate Roots

**Cause:** Roots exist but are in в„Ӯ \ в„қ (complex, non-real)

**From algebra:** Complex roots of real-coefficient polynomials come in conjugate pairs: a Вұ bi

**Example:** P(x) = xВІ - 2x + 5
- Discriminant: 4 - 20 = -16 < 0
- Roots: 1 Вұ 2i (complex conjugates)
- P(k) вү  0 for all integer k
- `totalZero = 0`, degree = 2 вҶ’ **undetermined**

**Detection criterion (quadratic):** Discriminant bВІ - 4ac < 0

**For higher degrees:** Factor theorem and numerical analysis needed

#### Category 4: Irrational Roots

**Cause:** Real roots exist but are not integers

**Example:** P(x) = xВІ - 2
- Roots: ВұвҲҡ2 вүҲ Вұ1.414...
- P(1) = -1 вү  0, P(2) = 2 вү  0
- No integer k gives P(k) = 0
- `totalZero = 0`, degree = 2 вҶ’ **undetermined**

**Rational Root Theorem:** If a polynomial with integer coefficients has a rational root p/q, then p divides the constant term and q divides the leading coefficient.

**Implication:** Most polynomials with arbitrary coefficients have irrational roots.

#### Category 5: Boundary Effects

**Edge case:** Root at exactly the boundary of the evaluation range may be missed depending on boundary handling (`<` vs `вүӨ`).

### Summary Table

| Category | Root Location | Example | Real? | Integer? | Detectable? |
|----------|---------------|---------|-------|----------|-------------|
| Outside range | x > range or x < -range | x = 500 | Yes | Yes | **No** |
| Positive definite | вҲ… (no real roots) | xВІ + 1 | No | No | **No** |
| Complex conjugate | a Вұ bi, b вү  0 | 1 Вұ 2i | No | No | **No** |
| Irrational | r вҲҲ в„қ \ в„Ө | вҲҡ2 | Yes | No | **No** |
| In range | -range/2 вүӨ x < range/2, x вҲҲ в„Ө | x = 5 | Yes | Yes | **Yes** |

### Distribution Implications

**For random polynomial coefficients:** Most polynomials are **undetermined** because:
1. Complex roots are common (especially for higher degrees)
2. Irrational roots are measure-theoretically "almost all" real roots
3. Integer roots are relatively rare in probability terms

**For structured coefficient exploration** (the Java code's approach):
- Coefficients are systematically varied via `setProductRange`
- This increases the likelihood of finding polynomials with integer roots
- The graph becomes enriched with determined polynomials

**The power set claim requires:** Generation parameters (`dimension`, `integerRange`, `setProductRange`) calibrated to ensure sufficient coverage of integer-root configurations.

### The Power Set Correspondence

**Key Claim:** For every element in the power set of possible root positions, there exists a graph node with corresponding root indexing.

| Power Set Element | Graph Node | Root Indexing |
|-------------------|------------|---------------|
| вҲ… (empty set) | Constant polynomial node | `muList = []`, `totalZero = 0` |
| {0} | Linear with root at 0 | `muList = [0]`, `totalZero = 1` |
| {0, 1} | Quadratic with roots at 0,1 | `muList = [0,1]`, `totalZero = 2` |
| {iвӮҒ, iвӮӮ, ..., iвӮ–} | Degree-k polynomial | `muList = [iвӮҒ, iвӮӮ, ..., iвӮ–]` |

**Size of Power Set:**
- `integerRange = R` вҶ’ R possible root positions
- Power set has 2^R elements
- Each element corresponds to a unique root configuration

### Binary Encoding of Root Patterns

Each determined polynomial's root pattern (`muList`) encodes as a **binary number**:

```
muList = [iвӮҒ, iвӮӮ, ..., iвӮ–]  
    вҶ“ ПҮбөў encoding (see NextBinary.java)
Binary string B  
    вҶ“ positional value (see ListToNumber.java)
Integer N
```

**Example (integerRange = 5, roots at positions 0, 2, 4):**
```
muList = [0, 2, 4]
Binary positions: bit 0, bit 2, bit 4 are set
Binary: 10101вӮӮ = 21вӮҒвӮҖ
```

### Link Prediction: Ordering Integers in the Graph

Given that each determined polynomial maps to an integer N via binary encoding:

**ML Task:** Predict `:NEXT_INTEGER` relationships between nodes whose binary encodings differ by 1.

```
Node(muList=[0])      вҶ’ N = 1   (binary: 1)
Node(muList=[1])      вҶ’ N = 2   (binary: 10)
Node(muList=[0,1])    вҶ’ N = 3   (binary: 11)
Node(muList=[2])      вҶ’ N = 4   (binary: 100)
...
```

**Link Prediction Discovers:**
```
N=1 в”Җв”Җ:NEXT_INTEGERв”Җв”Җ> N=2 в”Җв”Җ:NEXT_INTEGERв”Җв”Җ> N=3 в”Җв”Җ:NEXT_INTEGERв”Җв”Җ> N=4 ...
```

**Why This Is Meaningful:**
- The `:zMap` edges encode polynomial difference structure (structural, deterministic)
- The `:NEXT_INTEGER` edges encode integer ordering (emergent from binary encoding)
- ML discovers the integer ordering from polynomial root patterns

### ML Classification Task

**Input features:**
- `totalZero` вҖ” count of zeros found
- `d` вҖ” effectively `dimension - wNum` (the degree at this level)
- `n` вҖ” derived from `muList`
- `muList` вҖ” the zero positions
- Graph structure via `:zMap` embeddings

**Target:**
- `determined` вҲҲ {0, 1}

**Trivial solution?** The classification is **deterministic** post-hoc:
```
determined = (totalZero == d) ? 1 : 0
```

### Degree-Based Determinedness Probability

Empirical observation from graph queries reveals a clear pattern:

| Degree | wNum (for dim=4) | Typical determined | Reasoning |
|--------|------------------|-------------------|-----------|
| 0 | 4 | Always (trivial) | No roots needed |
| 1 | 3 | Almost always | Single root easy to find |
| 2 | 2 | Often | May have complex roots |
| 3 | 1 | Sometimes | Three roots needed |
| 4 | 0 | Rarely | Four integer roots unlikely |

**Key insight:** P(determined) decreases as degree increases. Only linear (degree 1) polynomials are "certainly determined" вҖ” higher degrees have probabilistic determinedness based on root locations.

See [algorithm_construction.md](../documentation/algorithm_construction.md) for how the constraint mechanism creates these patterns.

### Why ML Is Still Interesting

The "trivial" solution requires knowing `totalZero` вҖ” but this requires **evaluating the polynomial first**. The ML task is to predict **BEFORE** evaluation:

**The real ML challenge:** Given only coefficient patterns (vmResult) and graph position, predict whether evaluation WILL yield `totalZero == degree`.

1. **Predict without explicit `d`** вҖ” Can GraphSage embeddings infer degree from structure?

2. **Predict from partial information** вҖ” Given only `muList` and graph position, predict `determined`

3. **Predict `totalZero`** вҖ” This is harder: predict the count of zeros from other features

4. **Predict which roots are missing** вҖ” For undetermined nodes, where might the missing roots be?

### Code Insight: Why Some Are Undetermined

From `LoopList.java`, zeros are only counted if:

```java
// Only count zeros in positive half of range
if (listValueResult.compareTo(zero) == 0 && (x > 0)) {
    muList.add(x - caiIndex);
    caiIndex++;
}
if (listValueResult.compareTo(zero) == 0) {
    totalZero++;
}
```

**Causes of underdetermination:**

1. **Roots outside range:** Polynomial has roots at x = Вұ500 but `integerRange=200`

2. **Non-integer roots:** Polynomial has roots at x = 1.5, 2.7, etc.

3. **Complex roots:** Polynomial has roots in в„Ӯ \ в„қ

4. **Coefficient choice:** The `pArray` coefficients determine root locations

### Interesting Patterns

Certain polynomial families are more likely to be determined:

| Pattern | Determined? | Reason |
|---------|-------------|--------|
| Monic polynomials with small integer roots | Often yes | Roots within range |
| Polynomials with large leading coefficients | Often no | Roots compressed near origin or expanded beyond range |
| Polynomials from "nice" difference sequences | Often yes | Structured root patterns |
| Random coefficient polynomials | Usually no | Roots scattered, often irrational |

## Connecting Integers in Sequential Order

### The Sequencing Problem

Within each rational partition, integers must be connected in ascending order:

```
Partition for n/d:
  vвӮҒ вҶ’ vвӮӮ вҶ’ vвӮғ вҶ’ vвӮ„ вҶ’ ...

Where: vвӮҒ < vвӮӮ < vвӮғ < vвӮ„ < ...
```

### Link Prediction as Sequence Discovery

**The ML task:** Given nodes in a partition, predict edges that form the sequential chain.

```
Input:  {Node(5), Node(6), Node(10), Node(12)}  (unordered)
Output: Node(5) вҶ’ Node(6) вҶ’ Node(10) вҶ’ Node(12)  (sequential links)
```

### Why This Is Non-Trivial

The integer value is computed from `nBinary` via `ListToNumber`:

```java
// Integer reconstruction requires traversing the set list
for (int j = (loopIndex-1); j > -1; j--) {
    currentInt = (int)nList.get(j);
    currentPowerIndex = (whichPowerIndex-1) + (currentInt-1);
    currentResult = twoPower.pow(currentPowerIndex);
    resultInt = resultInt.add(currentResult);
}
```

The model must learn this implicit ordering from:
- `nBinary` structure
- `size` (number of set elements)
- `maxRange` (maximum element)

## Link Prediction Model Design

### Input Features

For each node, the model receives:

| Feature | Description | Derived From |
|---------|-------------|--------------|
| `n` | Numerator | Direct property |
| `d` | Denominator | Direct property |
| `nBinary` | Set list encoding | Direct property |
| `size` | Number of set elements | `len(nBinary)` |
| `maxRange` | Maximum set element | `max(nBinary)` |
| `value` | Integer representation | Computed via `ListToNumber` |

### Task 1: Partition Prediction (Classification)

**Objective:** Classify nodes into rational equivalence classes.

```
Model: f(node_features) вҶ’ (n, d) class label

Loss: Cross-entropy over (n,d) pairs
```

**Alternative:** Learn embeddings where same-rational nodes cluster together.

```
Model: g(node_features) вҶ’ embedding вҲҲ в„қбөҲ

Loss: Contrastive loss вҖ” same rational = close, different = far
```

### Task 2: Sequence Prediction (Link Prediction)

**Objective:** Predict `:NEXT_INTEGER` edges within partitions.

```
Model: h(node_i, node_j) вҶ’ P(node_i.value + 1 = node_j.value)

Loss: Binary cross-entropy
```

**Constraint:** Only predict within same partition (n, d match).

```python

    nodes = partition.nodes

        for j, node_j in enumerate(nodes):
            if i != j:
                pred = link_predictor(node_i.embedding, node_j.embedding)
                # pred should be high iff node_j.value = node_i.value + 1
```
