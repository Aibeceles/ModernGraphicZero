# Graph Structure Documentation

This document describes the Neo4j graph structure generated by the ZerosAndDifferences application and consumed by the GraphMLDifferences machine learning notebooks.

## Architecture Overview

### Multi-Threaded Database Access Pattern

The ZerosAndDifferences application employs a **producer-consumer pattern** where multiple computation threads share access to a single database writer thread:

```
┌─────────────────────────────────────────────────────────────────────┐
│                     PRODUCER THREADS                                 │
│                                                                      │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐               │
│  │ LoopsDriver  │  │ LoopListener │  │ LoopListener │  ...          │
│  │  (Worker 0)  │  │  (Worker 1)  │  │  (Worker N)  │               │
│  └──────┬───────┘  └──────┬───────┘  └──────┬───────┘               │
│         │                 │                 │                        │
│         └─────────────────┼─────────────────┘                        │
│                           ▼                                          │
│              ┌────────────────────────┐                              │
│              │   Shared Data Buffer   │  ← Semaphore-protected      │
│              │     (GaussBean1[])     │                              │
│              └───────────┬────────────┘                              │
│                          │                                           │
└──────────────────────────┼───────────────────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────────────────┐
│                    CONSUMER THREAD                                   │
│  ┌────────────────────────────────────────────────────────────────┐ │
│  │                      GaussTable1                                │ │
│  │                  (Database Writer)                              │ │
│  │                                                                 │ │
│  │  • Acquires data from buffer via semaphore                     │ │
│  │  • Executes Cypher queries to Neo4j                            │ │
│  │  • Manages JDBC connection lifecycle                           │ │
│  │  • Batches writes for performance                              │ │
│  └────────────────────────────────────────────────────────────────┘ │
│                          │                                           │
│                          ▼                                           │
│              ┌────────────────────────┐                              │
│              │      Neo4j Database    │                              │
│              │   (bolt://localhost)   │                              │
│              └────────────────────────┘                              │
└──────────────────────────────────────────────────────────────────────┘
```

### Synchronization Mechanism

All producer threads coordinate access to the shared buffer through:
- **Semaphores** (`tlSem`, `muSem`) - control buffer access permits
- **Synchronized methods** - ensure thread-safe database operations
- **Buffer notifications** - wake consumers when data is available

## Graph Dimensions and Width

The graph's structure is controlled by three primary configuration parameters that define both global width and node-level branching behavior.

### Configuration Parameters

| Parameter | Example Value | Description |
|-----------|---------------|-------------|
| `dimension` | 5 | Polynomial degree - determines the tree **depth** (depth = dimension + 1 levels) |
| `integerRange` | 200 | Evaluation range - polynomials evaluated from `-integerRange/2` to `+integerRange/2` (e.g., -100 to +99) |
| `setProductRange` | 4 | Coefficient iteration range - controls how many polynomial variants are explored |

### Global Graph Width

The total width of the graph is defined by the product of:

```
Total Source Polynomials = (setProductRange)^dimension × coefficient_combinations

Example: setProductRange=4, dimension=5
         → Up to 4^5 = 1024 source polynomial variants
```

**Horizontal expansion factors:**

1. **Source Polynomial Count:** `setProductRange^dimension` potential starting polynomials
2. **Nodes Per Polynomial Chain:** `dimension + 1` nodes (one per difference level)
3. **Evaluation Sample Width:** `integerRange` sample points per polynomial evaluation

```
┌──────────────────────────────────────────────────────────────────────────────┐
│                        GLOBAL GRAPH WIDTH VISUALIZATION                       │
│                                                                              │
│  setProductRange = 4, dimension = 3, integerRange = 200                      │
│                                                                              │
│  Level 0 (d=3):  [P₁][P₂][P₃][P₄] ... [P₆₄]    ← 4³ = 64 source polynomials │
│                    │   │   │   │         │                                    │
│  Level 1 (d=2):   [△₁][△₂][△₃][△₄]...[△₆₄]    ← 1st differences            │
│                    │   │   │   │         │                                    │
│  Level 2 (d=1):   [△²₁][△²₂][△²₃]...[△²₆₄]    ← 2nd differences            │
│                    │   │   │   │         │                                    │
│  Level 3 (d=0):   [C₁][C₂][C₃][C₄] ... [C₆₄]   ← Constants (many merge)     │
│                                                                              │
└──────────────────────────────────────────────────────────────────────────────┘
```

### Node-Level Width (Per-Node Branching)

From any single node's perspective, width is determined by:

#### 1. **Child Count (Outgoing Relationships)**

Each DifferencePolynomial node connects to exactly **one child** via `:zMap` in the standard difference chain. However, the Java application may produce **multiple paths** through the same node when different source polynomials share common difference subsequences.

```java
// From LoopList.java - each workNum produces one difference sequence
for (int y = 0; y < integerRange; y++) {
    arg1 = ((LoopList)rListB.get(x)).get(y);
    arg2 = ((LoopList)rListB.get(x)).get(y+1);
    listResult = arg2.subtract(arg1);  // Single difference → single child
}
```

#### 2. **Parent Count (Incoming Relationships)**

A node may have **multiple parents** if the same interpolated polynomial identity (`vmResult`) is reached from different source polynomials:

```
                [Source P₁]        [Source P₂]
                     │                  │
                     ▼                  ▼
                [Diff A]            [Diff B]
                     │                  │
                     └────────┬─────────┘
                              ▼
                    [Shared vmResult X]  ← Node with 2 parents
```

#### 3. **Sibling Width at Each Level**

The number of sibling nodes (nodes at the same `d` value) is approximately:

```
Siblings at depth d ≈ setProductRange^dimension × uniqueness_factor

Where uniqueness_factor decreases toward d=0 (constants tend to merge)
```

### Width Progression Through Difference Levels

As differences are computed, the width **converges** toward lower levels:

| Level | d | Description | Typical Width Behavior |
|-------|---|-------------|----------------------|
| 0 | `dimension` | Source polynomials | `setProductRange^dimension` unique nodes |
| 1 | `dimension-1` | 1st differences | Slightly fewer (some merging) |
| 2 | `dimension-2` | 2nd differences | More merging occurs |
| ... | ... | ... | ... |
| `dimension` | 0 | Constants | Heavy merging - many paths lead to same constant |

### Worker Number (`wNum`) and Depth Relationship

The `wNum` (worker number) in the Java code corresponds directly to the **difference iteration level**, which is the inverse of the `d` (depth) property stored in Neo4j:

```
wNum = 0  →  d = dimension    (source polynomial, no differences taken)
wNum = 1  →  d = dimension-1  (1st difference)
wNum = 2  →  d = dimension-2  (2nd difference)
...
wNum = dimension  →  d = 0    (constant, final difference)
```

**Width shrinkage per level:**

Each difference operation reduces the data width by 1:

```java
// From LoopList.subsequentWorkerListInit()
for (int y = 0; y < integerRange; y++) {
    if (y + 1 < integerRange) {
        arg2 = ((LoopList)rListB.get(x)).get(y + 1);
    } else {
        arg2 = zero;  // Edge case: last element
    }
    listResult = arg2.subtract(arg1);
}
```

| wNum | Data Points per LoopList | Effective Width |
|------|-------------------------|-----------------|
| 0 | `integerRange` | 200 |
| 1 | `integerRange - 1` | 199 |
| 2 | `integerRange - 2` | 198 |
| ... | ... | ... |
| dimension | `integerRange - dimension` | 195 (for dim=5) |

### Width Formula Summary

```
Global Width = setProductRange^dimension × branching_factor
Node Data Width (wNum=n) = integerRange - n
Sibling Count at depth d = f(setProductRange, dimension, merging_rate)
```

Where `merging_rate` increases toward `d=0` due to many polynomial difference sequences converging to the same constants.

### Width in the `integerRange` Dimension

Each polynomial is evaluated at `integerRange` integer points. The width of evaluation determines:

- **Zero detection range:** Roots can only be found within `[-integerRange/2, +integerRange/2)`
- **Sample density:** Higher `integerRange` → finer sampling → more zeros detected
- **Memory footprint:** Each `LoopList` contains `integerRange` values

```java
// From LoopList.java - evaluation loop width
int halfIntegerRange = integerRange / 2;
for (int x = -halfIntegerRange; x < halfIntegerRange; x++) {
    // Evaluate polynomial at x
    // Results populate the LoopList
}
```

### Configuration Impact on Graph Size

| Configuration | Graph Impact |
|--------------|--------------|
| `dimension ↑` | Deeper tree, exponentially more source combinations |
| `setProductRange ↑` | Wider at each level, multiplicatively more nodes |
| `integerRange ↑` | Better zero detection, larger memory per node |

**Practical example from codebase defaults:**

```java
// From LoopsDriverTwoPManager.java
int dimension = 5;
int setProductRange = 4;
int integerRange = 200;

// Results in:
// - Tree depth: 6 levels (d=5 down to d=0)
// - Theoretical max source polynomials: 4^5 = 1024
// - Evaluation points per polynomial: 200 (-100 to +99)
// - Memory per LoopList: ~200 BigDecimal values
```

## Node Types

### 1. DifferencePolynomial Node (`:Dnode`)

The primary node type representing a polynomial derived from the difference analysis.

| Property | Type | Description |
|----------|------|-------------|
| `vmResult` | String | Monomial coefficients (Newton interpolation primary) identifying this difference polynomial |
| `n` | Integer | **Numerator** - represents the rational number component in the polynomial's fractional representation |
| `d` | Integer | **Denominator/Depth** - the degree level in the difference tree (0 = constant, 1 = linear, 2 = quadratic, etc.) |
| `totalZero` | Integer | Count of zero evaluations - how many integer inputs evaluate to zero |
| `muList` | String | List of **μ (mu) values** - the indices where the polynomial evaluates to zero |
| `determined` | Integer | **Classification flag**: `1` = fully determined (zeros equal degree), `0` = underdetermined |

**Semantic Meaning:**
- A DifferencePolynomial with `d=3` and `totalZero=3` is "determined" (determined=1)
- A DifferencePolynomial with `d=3` and `totalZero=2` is "underdetermined" (determined=0)

### 2. Origin Node (`:CreatedBy`)

Metadata node tracking the computational origin of each polynomial.

| Property | Type | Description |
|----------|------|-------------|
| `resultId` | Integer | **Batch correlation ID** - groups related polynomials from the same computational pass |
| `wNum` | Integer | **Worker number** - which difference iteration level (0 = source polynomial, 1+ = differences) |
| `pArray` | String | **Polynomial array state** - the coefficient array at computation time |
| `skipList` | String | **Mu mask list** - indices skipped during iteration |
| `translateIndex` | Integer | Translation offset for polynomial evaluation |
| `pArrayIndexValue` | Integer | Index value in the polynomial coefficient array |

### 3. Evaluation Node (`:Evaluate`)

Represents specific evaluation points and their properties.

| Property | Type | Description |
|----------|------|-------------|
| `Value` | String | The evaluation index (x-value) |
| `n` | String | Numerator of the μ representation |
| `d` | String | Denominator of the μ representation |
| `nBinary` | String | Binary representation of the evaluation pattern |
| `size` | Integer | Size of the binary pattern |
| `centerIndex` | Integer | Center index in the pattern |
| `verticalTranslateIndex` | Integer | Vertical translation offset |

### 4. μ-Number Node (`:MuNumber`)

Represents the μ (mu) rational numbers used in the analysis.

| Property | Type | Description |
|----------|------|-------------|
| `Value` | String | The μ value |
| `n` | String | Numerator |
| `d` | String | Denominator |
| `nBinary` | String | Binary representation |
| `size` | Integer | Binary pattern size |
| `maxRange` | Integer | Maximum range of the pattern |

### 5. Configuration Node (`:Configure`)

Stores runtime configuration parameters.

| Property | Type | Description |
|----------|------|-------------|
| `setProductRange` | Integer | Range for set product iteration |
| `setProductRangeIncrement` | Integer | Increment step |
| `maxSetProductRange` | Integer | Maximum iteration bound |
| `dimension` | Integer | Polynomial dimension/degree |
| `maxFigPScalar` | Integer | Maximum figure polynomial scalar |
| `integerRange` | Integer | Integer evaluation range |
| `pArray` | String | Initial polynomial array |

## Relationship Types

### 1. ZeroMapping (`:zMap`)

Connects DifferencePolynomials that share a computational relationship through the zero-mapping process.

```
(DifferencePolynomial)-[:zMap]->(DifferencePolynomial)
```

**Semantics:**
- Links a polynomial to its next-level difference
- Creates a **tree structure** from source polynomial down to constant difference
- Undirected in GDS projections for graph algorithms

### 2. CreationLink (`:CreatedBye`)

Connects a DifferencePolynomial to its Origin metadata.

```
(DifferencePolynomial)-[:CreatedBye]->(Origin)
```

**Semantics:**
- Tracks provenance of each polynomial
- Enables filtering by computational batch
- Links worker output to computation context

### 3. EvaluationLink (`:Envalues`)

Connects a DifferencePolynomial to its zero-evaluation points.

```
(DifferencePolynomial)-[:Envalues]->(Evaluation)
```

**Semantics:**
- Records which integer values make the polynomial zero
- Used for pattern analysis of zero distributions

### 4. TwoPolynomialLink (`:TwoPolynomial`)

Links for two-polynomial comparative analysis.

```
(DifferencePolynomial)-[:TwoPolynomial]->(Evaluation)
```

### Node Classification Task

Using the graph structure for classification:

**Target:** Predict `determined` property (0 or 1)

**Features:**
- `totalZero` - count of rational roots found
- `d` - polynomial degree/depth level
- `n` - numerator identifier
- Graph-based features via GraphSage embeddings

### Link Prediction Task

Using embeddings to predict missing `:zMap` relationships:

**Input:** GraphSage node embeddings from `d` property
**Output:** Probability of `:zMap` connection between node pairs

## Cypher Query Examples

### Get All DifferencePolynomials with Their Depths
```cypher
MATCH (p:Dnode)
RETURN p.vmResult AS polynomial, 
       p.d AS depth, 
       p.totalZero AS zeros,
       p.determined AS isDetermined
ORDER BY p.d
```

### Find Determined vs Underdetermined Polynomials
```cypher
MATCH (p:Dnode)
WHERE p.determined = 1
RETURN COUNT(p) AS determinedCount

UNION

MATCH (p:Dnode)  
WHERE p.determined = 0
RETURN COUNT(p) AS undeterminedCount
```

### Traverse the Difference Tree
```cypher
MATCH path = (root:Dnode {d: 0})-[:zMap*]->(leaf:Dnode)
WHERE NOT (leaf)-[:zMap]->()
RETURN path
```

### Get Polynomials by Computational Batch
```cypher
MATCH (p:Dnode)-[:CreatedBye]->(origin:CreatedBy {resultId: 42})
RETURN p, origin
```

2. **Processing:** Iterated differences computed by worker threads

5. **Output:** Graph structure ready for GDS algorithms and ML

## Width in Database Operations

### How Width is Encoded in Cypher

The `GaussTable1` class batches writes and creates chains of nodes with their `:zMap` relationships in a single operation:

```cypher
-- From GaussTable1.ggoCCypher (simplified)
WITH vmResult, n, d, z, m AS arrays
UNWIND RANGE(0, SIZE(vmResult) - 1) AS idx
WITH vmResult[idx], n[idx], d[idx], z[idx], m[idx]
MERGE (dNode:Dnode {vmResult: vmResult, n: n})
ON CREATE SET dNode.totalZeros = z, dNode.d = d, ...
WITH collect(dNode) AS nodeList
UNWIND range(0, size(nodeList)-2) AS index
WITH nodeList[index] AS source, nodeList[index+1] AS target
MERGE (source)-[:zMap]->(target)
```

**Key insight:** Each batch write creates a **chain** of `dimension+1` nodes connected sequentially. The width of the overall graph emerges from multiple batches creating parallel chains that may share nodes via `MERGE`.

### Measuring Graph Width with Cypher

```cypher
-- Count nodes at each depth level (width per level)
MATCH (p:Dnode)
RETURN p.d AS depthLevel, COUNT(p) AS nodeCount
ORDER BY p.d DESC

-- Find the widest level
MATCH (p:Dnode)
WITH p.d AS level, COUNT(p) AS width
RETURN level, width
ORDER BY width DESC
LIMIT 1

-- Count total relationships (total graph connectivity)
MATCH ()-[r:zMap]->()
RETURN COUNT(r) AS totalConnections

-- Find nodes with multiple incoming relationships (convergence points)
MATCH (child:Dnode)<-[r:zMap]-()
WITH child, COUNT(r) AS incomingCount
WHERE incomingCount > 1
RETURN child.vmResult, child.d, incomingCount
ORDER BY incomingCount DESC
```

### Width vs. Connectivity Trade-offs

| Metric | Low Width Configuration | High Width Configuration |
|--------|------------------------|-------------------------|
| `setProductRange=2, dimension=3` | 8 source polynomials | Sparse graph |
| `setProductRange=4, dimension=5` | 1024 source polynomials | Dense graph |
| Memory usage | Lower | Higher |
| Training data volume | Limited | Abundant |
| Node merging | Less | More convergence at low `d` |
